# 企业微信(WeCom)操作手册 (V2.0)

供 ReAct Agent 在每次决策时参考。内容偏**结构与位置**,而非具体文字。

---

## 1. 页面分类与状态判定

| page 值 | 含义 | 关键特征 |
|---------|------|------|
| `HOME` | 主页/会话列表页 | 包含会话列表，用于选择目标联系人。 |
| `CHAT` | 单个会话聊天页 | 顶部包含联系人信息，底部包含输入栏。 |
| `UNKNOWN` | 其它/非聊天页 | 相册选择器 (picker)、图片预览、搜索结果、设置页等。 |

> **核心判定原则**: 任何状态的判定，**优先看 `page` 字段**。如果 `page=CHAT`，但无法找到可编辑的输入框，则判定为**非文本输入模式**。
> `page=UNKNOWN` 不一定是坏状态：相册、图片预览、发送确认页都可能显示为 UNKNOWN。只要 UI tree 中出现 `发送` / `完成` / 勾选框 / 预览大图，就继续完成媒体发送流程，不要因为节点少就放弃。

## 2. 聊天页 (page=CHAT) 核心流程

### 2.1 目标会话进入 (Role-Based Navigation)
*   **直接路径**: 当前 `page=HOME`，在消息列表中的目标联系人列表项，应使用 `role=chat_target` 命中缓存并点击。
*   **搜索路径**: 如果列表页找不到，需点击顶栏的搜索入口，输入关键词，然后点击搜索结果中的第一个匹配项。
*   **判定成功**: 进入 `page=CHAT`，且屏幕顶部区域（约占 15%）的节点描述与目标联系人匹配，即成功。

### 2.2 文本输入模式切换与输入 (关键模式)
底部输入栏从左到右的典型结构为：
`[ 模式切换图标 ] [ 中间输入区 ] [ 表情图标 ] [ 附件/更多图标 (+) ]`

> **重要：表情 vs 附件入口**
> 在聊天输入栏右侧通常有两个相邻的 icon-only ImageView：左边是**表情/笑脸**，右边最末尾才是**附件/更多 `+`**。
> 发送图片、GIF、文件时，`compose_plus` 必须选择输入栏同一行里**最右侧**的图标；不要点它左边的笑脸图标。
> 若点完后出现大量 `[微笑]`、`[撇嘴]`、`[色]` 这类表情文本，说明误入表情面板；此时原来的笑脸位置通常会变成**键盘图标**，但它仍然只是“表情/键盘切换”，不是附件入口。下一步不要继续点表情格子，也不要点键盘图标，应改点输入栏最右侧的 `+`，或先 back 收起后重新点最右侧 `+`。

**A. 判定输入模式：**
1.  **文本模式 (Text Mode)**: 底部输入栏包含一个 `editable=true` 的 EditText 节点。
2.  **非文本模式 (Non-Text Mode)**: `page=CHAT`，但 UI tree 中找不到任何 `editable=true` 的节点。

**B. 模式切换流程 (当处于非文本模式时)：**
1.  **操作**: 必须点击底部输入栏最左侧的**模式切换图标**（通常是第一个 ImageView）。
2.  **目的**: 强制切换输入模式，使下一次 UI dump 能够捕获到 `editable=true` 的 EditText 节点。
3.  **输入**: 切换成功后，再执行 `input_text(role=message_input)`。

### 2.3 发送消息 (Send Message)
1.  **输入**: 确保已在文本模式，执行 `input_text(role=message_input)` 写入内容。
2.  **发送**: 点击底部输入栏右侧的发送按钮（`role=send_button`）。
3.  **成功判定**:
    *   **最可靠信号**: `page` 从 `UNKNOWN` (如预览页) 或 `CHAT` 切回 `CHAT`，且输入框内容被清空或恢复占位符。
    *   **次要信号**: 聊天区底部出现一个新的、右对齐的消息气泡。
    *   **注意**: 成功发送后，应立即 `done(success=true)`，避免重复发送。

## 3. 附件与图片选择器 (page=UNKNOWN)

### 3.1 附件入口
*   **路径**: 点击输入栏右侧最末尾的附件/更多 `+` 图标（`role=compose_plus`）。
*   **禁止误点**: `+` 左边紧挨着的笑脸/表情图标不是附件入口；点它会打开表情面板。进入表情面板后，该位置可能变成键盘图标，但仍然不是附件入口。
*   **结构判定**: 如果 UI tree 中可见输入框 `EditText`，在它右侧同一行可能有两个 `ImageView`；取更靠右、通常也是输入栏行内最后一个的节点作为 `compose_plus`。
*   **正确结果**: 屏幕下半部出现附件网格，通常包含 `图片`、`拍摄`、`收藏`、`语音通话`、`红包`、`文档` 等 label。
*   **错误结果**: 出现大量 `[微笑]` / `[撇嘴]` / `[色]` 这类表情 label，表示点到了表情入口。恢复方式：不要选择任何表情，不要点键盘切换图标；点击输入栏最右侧 `+` 或 back 后重新点击最右侧 `+`。

### 3.2 相册选择器 (Picker)
*   **入口**: 在网格面板中，寻找带有“图片/相册/照片”标签的格子，点击进入相册。
*   **结构**: 主体是矩形格子网格，按 UI tree 顺序排列。
*   **识别真实图片/GIF**: 真实缩略图的格子内通常是**纯 ImageView 或 ImageView + 选择框/勾选覆盖层**，没有“拍摄/图片/相册”等 TextView 标签。快捷入口、相机入口、文件夹入口会带文字 label，不能当作目标缩略图。
*   **最新文件优先**: 刚刚落入 `Pictures/WeComAgent/` 的图片/GIF 通常排在相册网格最前面；若第一格是纯缩略图或带勾选框的缩略图，就优先选择它。
*   **选择流程**:
    1.  点击目标缩略图，右上角出现勾选标记(✓)。
    2.  **关键**: 必须点击右下角的**“发送/Send”**按钮（或“完成”按钮），才能将图片发送到聊天页。
    3.  **成功判定**: `page` 从 `UNKNOWN`(picker/预览)切回 **`CHAT`**。

### 3.3 图片/GIF 预览与确认页
选中缩略图后，企业微信可能进入一个很小的 `page=UNKNOWN` 预览/确认页。这个页面的 UI tree 可能只有十来个节点，但通常包含：
*   一个大图/预览 `ImageView`
*   一个 `CheckBox` 或勾选控件
*   一个文字为 `发送` 的 TextView/Button，常在右下或底部区域

这是正常的发送确认页，不是错误页面。下一步必须点击 `发送`，并标记 `_locator_role="gallery_send_button"`。点击后若返回 `page=CHAT`，立即 `done(success=true)`，不要再次进入相册或重复选择图片。

点击 `gallery_send_button` 后，如果页面短时间仍停留在 `UNKNOWN` / 预览 / loading 状态，不要按 `back`。这通常表示媒体仍在上传或企业微信正在处理发送；应等待下一次 UI 刷新或使用 `wait_ui` 观察页面回到 `CHAT`。按 `back` 可能会取消发送并退回会话列表，随后误入其它聊天。

### 3.4 成功判定的**严格条件**

**只看 `page=CHAT` 不够**:点选不同的会话也会让 page=CHAT。真正的发送成功要同时满足三个条件:
1. 上一步是 `tap_node`(不是 `back`、`swipe`、`home`)
2. 上一步的 `before_page` 是非 CHAT(picker / UNKNOWN / 预览页)
3. 当前 CHAT 页的**顶栏文字**与媒体阶段一开始你所在聊天的顶栏文字**一致**(同一个对象)

后端会自动检查这三条并通过 `[验证]` 注解给出结论:
- "页面已从 picker 回到原聊天主页 → 发送已生效" → 立刻 `done(success=true)`
- "已落到 CHAT 页但顶栏是「X」而非起始的「Y」" → **千万不要** `done(success=true)`。这是 wrong-chat,说明你之前 back 离开过 picker 又点进了别的会话。处理:先 back 回会话列表,重新打开 `「Y」`,核实是否需要重发(看新的聊天历史里是否真的有那张图)
- "已点击媒体发送按钮但页面仍在 picker/预览/上传状态" → 静等,下一步用 `wait_ui` 或重新 dump,**不要** back

### 3.4 媒体发送的推荐顺序
1.  在聊天输入栏点击最右侧 `+`，role=`compose_plus`。
2.  在附件网格点击 `图片` / `相册` 入口，role=`media_picker_entry`。
3.  在相册网格点击最新的真实缩略图/GIF，role=`gallery_first_item`。
4.  如果出现预览/确认页，点击 `发送` / `完成`，role=`gallery_send_button`。
5.  点击发送后如果还在 `UNKNOWN`，等待；不要 back，不要重新进其它会话。
6.  观察到页面回到 `CHAT` 后立即成功结束。

## 4. 文本输入与发送的通用流程总结

| 步骤 | 目标 | 结构信号/角色 | 备注 |
| :--- | :--- | :--- | :--- |
| **1. 导航** | 进入目标会话 | `role=chat_target` | 优先使用缓存的会话列表定位。 |
| **2. 模式检查** | 确保处于文本输入模式 | 检查是否存在 `editable=true` 节点 | 如果不存在，必须先点击底部输入栏最左侧的模式切换图标。 |
| **3. 输入文本** | 写入消息内容 | `role=message_input` | 写入目标文本。 |
| **4. 发送** | 发送消息 | `role=send_button` | 触发发送动作。 |

## 5. 常见错点与恢复模式

| 现象 | 原因分析 | 推荐处理 |
| :--- | :--- | :--- |
| `page` 未变，UI tree 相同 | 点击的节点是无效的或不可交互的。 | 换行或换列，尝试点击该组相邻的另一个节点。 |
| 页面跳转到非预期状态 | 误触了表情、位置或附件等子面板。 | 使用 `back` 回退到上一级页面，再重新尝试目标节点。 |
| 出现大量 `[微笑]` / `[撇嘴]` 表情 label | 误点了笑脸/表情入口。 | 不要点表情或键盘图标，改点输入栏最右侧 `+`；必要时先 back 收起面板。 |
| `UNKNOWN` 页面节点很少但有 `发送` | 已进入图片/GIF 预览确认页。 | 点击 `发送`，role=`gallery_send_button`；回到 CHAT 后 done。 |
| 点击 `发送` 后仍在 `UNKNOWN` / loading | 媒体上传或发送处理中。 | 等待 UI 回到 CHAT；不要 back，否则可能取消发送并跳到会话列表。 |
| 连续 3 次操作失败 | 系统判定操作序列异常或卡死。 | 停止操作，等待系统自动重置或退出当前流程。 |
| 文本输入框无法输入 | 处于非文本模式。 | 参照 2.2 流程，点击底部输入栏最左侧的模式切换图标。 |

## 6. 关于 `_locator_role` 标注

为了让缓存学习生效，在每次可复用的 tap 上，在 `args` 里加 `_locator_role`，取值见 system prompt 中的列表。
**结构特征**(view_id / cls / bounds_ratio)由后端从节点抽取，你只需选对 role 标签。
后续同类目标会优先命中缓存，跳过 LLM 直接执行。
