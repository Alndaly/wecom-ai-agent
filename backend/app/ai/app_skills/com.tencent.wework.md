# 企业微信(WeCom)操作手册

供 ReAct Agent 在每次决策时参考。内容偏**结构与位置**,而非具体文字 ——
企微跨版本、跨 ROM 措辞会变,但布局、节点类型和页面跳转关系是稳定的。

---

## 1. 页面分类(UI tree 首行 `page=` 字段)

| page 值 | 含义 |
|---------|------|
| `HOME` | 消息/通讯录/工作台 等主 tab,可能包含会话列表 |
| `CHAT` | 与单个联系人的聊天页(顶栏=联系人名,底部=输入栏) |
| `UNKNOWN` | 其它:相册选图、图片预览、搜索结果、弹窗、设置页等 |

> 任何"是否到达 X 状态"的判定,**优先看 page 字段**,比扫描具体文字鲁棒。

---

## 2. CHAT 页输入栏结构(最常踩坑的地方)

底部一行,从左到右典型排列:

```
[ 模式切换 ImageView ] [ 中间输入区 ] [ 表情 ImageView ] [ 附件/更多 ImageView (+) ]
```

**中间输入区有两种互斥状态**:

- **键盘文本模式**: 含一个 `editable=true` 的 EditText 节点 —— 可直接 `input_text`
- **语音/录音模式**: 是个普通 Button/View(常显示"按住...说话"之类,字样不可靠),
  **没有 editable 节点**

### 判定与恢复
- `page=CHAT` 且 UI tree 找不到任何 `editable=true` 节点 → 处于**非文本模式**
- 切换方法:按 UI tree 顺序点底部一组并排 ImageView 里**最左侧那一个**(模式切换图标),切换后下一次 dump 应该能看到 EditText
- 点错弹出其它面板(表情/位置/...)→ 用 `back` 退出,再试该组下一个节点

---

## 3. + 附件面板

- 入口:输入栏右侧最末一个 ImageView(通常是 + 或更多图标)
- 点开后,屏幕下半部出现**网格面板**,每格是 RelativeLayout 内含 ImageView 图标 + TextView 标签
- 网格里**带"图片/相册/照片"标签**的格子是相册入口
- 进入相册后 page 变成 `UNKNOWN`(picker)

### 点错恢复
- 点了输入栏右侧 ImageView 但下半屏没出现网格面板,反而是单一表情网格/语音卡片 → 点错了,`back` 回 CHAT 再换该组下一个 ImageView

---

## 4. 相册图片选择器(picker,page=UNKNOWN)

- 顶栏标题:"图片和视频"或类似
- 主体是**矩形格子网格**,按 UI tree 顺序排列

### 跳过工具入口
- **第一格(或前几格)常是工具入口**:格子内含 TextView 子节点("扫描文档"/"拍摄"/"相机"等)
- **真实图片缩略图**: 格子内**只有 ImageView,没有 TextView 子节点**
- 最新存入的图片排在前几个真实缩略图位置(按时间倒序)

### 选图 → 发送
- 点击缩略图后,右上角出现勾选标记(✓)
- **勾选 ≠ 发送**: 必须再点右下角**"发送/Send"** 按钮,未选图时它是禁用灰色态
- 部分版本点缩略图后会跳进**预览/编辑页**(占满屏的大图):右下角同样有"发送"按钮,按同理点一下即可

### 发送图片成功的唯一可靠信号
- page 从 `UNKNOWN`(picker/预览)切回 **`CHAT`** → 发送已生效
- 一旦观察到这个 page 切换,**立即 `done(success=true)`**,不要再操作
  (重复操作会导致同张图被发多次)

---

## 5. 打开特定联系人聊天

### 直接路径(常用)
1. 当前 `page=HOME` 且消息列表里有目标联系人的列表项 → 直接 tap 该列表项
2. 进入 `page=CHAT`,顶栏(屏幕顶部 ~15% 区域内的 TextView)显示联系人名即成功

### 搜索路径
- HOME 列表里没看到目标 → tap 顶栏的搜索入口(放大镜图标或"搜索"占位条)
- 输入联系人名 → tap 搜索结果中第一个匹配项 → 进入 CHAT

### "已经在目标聊天"的判定
- `page=CHAT` **且** 屏幕顶部 ~15% 区域内有节点 text/desc == 目标联系人名 → 已在目标聊天
- 此时**不要**再 tap 任何 "chat_target" 节点(只会点到标题栏自己,无效;后端会判 stuck)

---

## 6. 发送文本

- 在 CHAT 页且 EditText 存在 → `input_text` 写入内容 → tap 发送按钮
- 成功判定:
  - 输入框 EditText 的 text 变回空字符串或占位符,**或**
  - 聊天区底部出现一个新的右对齐消息气泡,文本和刚发送的一致

---

## 7. 常见错点恢复模式

| 现象 | 处理 |
|------|------|
| tap 后 page 没变,UI tree 大致相同 | 点击是无效的,换同行另一个节点 |
| tap 后切到了非预期页面 | `back` 回退,再换节点 |
| 连续 3 次相同 tap 都没 page 切换 | 后端会自动判 stuck 并退出 |
| 发了 input_text 但 EditText 内容没变 | 节点不是真正的输入框,LLM 重新挑可编辑节点 |
| picker/预览页一直退不回 CHAT | 看是否有需要先确认的浮层(右上角 ✓ 或右下角"完成") |

---

## 8. 关于 `_locator_role` 标注

为了让缓存学习生效,在每次可复用的 tap 上,在 args 里加 `_locator_role`,取值见 system prompt 中的列表。
**结构特征**(view_id / cls / bounds_ratio)由后端从节点抽取,你只需选对 role 标签。
后续同类目标会优先命中缓存,跳过 LLM 直接执行。
